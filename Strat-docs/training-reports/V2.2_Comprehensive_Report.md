## 1. Background & Motivation

### The V2.2 Hypothesis

V2 (Qwen3-8B, Opus-distilled) established a strong baseline at MAE 1.544 on the Opus holdout. V2.1 attempted to improve via chain-of-thought training but catastrophically regressed to MAE 3.244. The post-mortem identified two paths forward:

1. **Expand article diversity** — V2 trained on only 813 articles; 17,260 remained untapped in the production database
2. **Upgrade base model** — Qwen3.5-9B showed MAE 1.297 on the holdout *without any fine-tuning*, already 16% better than V2's fine-tuned Qwen3-8B

V2.2 pursued both simultaneously while maintaining the proven short-reason training format (no think_text/CoT).

### What Changed from V2

| Dimension | V2 | V2.2 |
|-----------|-----|------|
| Base model | Qwen3-8B (8.0B params) | Qwen3.5-9B (9.0B params) |
| Teacher model | Claude Opus ($50-70) | Gemini 3 Flash ($0-6) |
| Training articles | 813 | ~2,000 (expanded from DB) |
| Training examples | ~5,000 | 11,785 |
| Eval examples | ~500 | 1,296 |
| Profiles | 30 (Opus-scored) | 45 (25 Gemini + 20 inherited) |
| Training format | Short reason | Short reason (same) |
| Think_text | Not used | Not used |
| Cost | ~$60-80 (Opus scoring) | ~$6 (Gemini scoring) |

## 9. Incident: OOM During Export

### What Happened
Training completed successfully at 03:20. Phase 3 (eval + GGUF export) ran the bulk evaluation, then attempted `model.cpu()` to save the merged model. The training process was already using 29GB RAM (model + residual optimizer state), and moving the 18GB model to CPU RAM pushed total to 72GB virtual — triggering the Linux OOM killer at 04:20.

### Impact
- Training results: **Not lost** (adapters saved at 03:20, before the crash)
- Bulk eval results: **Not lost** (completed before crash)
- GGUF export: **Failed** (only config files written)

### Resolution
Created `merge_export_v22.py` — a lightweight standalone script that loads base model + adapters in a fresh process with `device_map="cpu"` and `low_cpu_mem_usage=True`. Peak RAM: ~18GB (within 32GB system RAM). Script succeeded on second run.

### Additional Fix
llama.cpp's `convert_hf_to_gguf.py` didn't recognize `Qwen3_5ForCausalLM`. Fixed by `git pull` — latest version added support.

### Lesson
For 9B+ models on 32GB RAM systems, always use a separate merge/export process — never attempt `model.cpu()` inside the training process where optimizer states may still occupy RAM.

## 11. Known Limitations & Future Work

### Limitations
1. **Critical band sample size** — Only 9 critical-band examples in holdout. The +13% MAE regression may not be real.
2. **Computer Engineering profile** — Consistently worst profile (MAE 1.77-2.07). May need targeted training data.
3. **Gemini-Opus calibration gap** — Model learned Gemini's scale (MAE 0.848 vs teacher) but Gemini itself has 0.893 MAE vs Opus. Some of V2.2's Opus holdout error is inherited from teacher inaccuracy.
4. **Moderate/High band** — Still the weakest bands (MAE 1.7). These are intrinsically hard — borderline relevance is subjective.

### Recommended Next Steps
1. **Production monitoring** — Run real scans and compare user satisfaction to V2 era
2. **Address weak profiles** — Collect targeted data for computer engineering, corporate law, undeclared students
3. **Isotonic calibration** — Post-hoc calibration at inference time could reduce the Gemini→Opus scale gap
4. **Remove Gemini scoring cron** — Scoring is complete; cron job at 10:15 AM KWT should be removed
5. **V2.3 planning** — Consider: multi-epoch training (now safe with 2000+ articles), synthetic hard negatives for moderate band, or Opus-Gemini hybrid scoring
