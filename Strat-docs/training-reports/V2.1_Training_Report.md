## 1. What Is StratOS?

StratOS is a personalized strategic intelligence dashboard. It fetches news from multiple sources (RSS, DuckDuckGo, Serper/Google), scores them for relevance using a local Ollama-hosted LLM, and presents a prioritized feed via a web UI.

**Project root:** `~/Downloads/StratOS/StratOS1/`
**Frontend:** vanilla JS (app.js, market.js, mobile.js, settings.js), served from `frontend/`
**Backend:** Python (`backend/`), SQLite (`backend/strat_os.db`), YAML profiles (`backend/profiles/`)
**Server:** `cd backend && python3 main.py --serve --background` on port 8080

### The Scorer's Role
The **scorer** is the core intelligence layer. During a scan (~300 articles), each article is scored 0-10 for relevance to the user's profile (role, location, interests, tracked companies). This must happen in <5 minutes — speed matters as much as accuracy.

**Inference flow:**
```
User profile → system prompt
Article → user prompt
Model outputs → SCORE: X.X | REASON: brief explanation
```

The scorer runs as a fine-tuned Qwen 3 8B model served via Ollama. The model is trained through a distillation loop: Claude Opus scores articles as ground truth, disagreements become training data, and DoRA fine-tuning updates the local model.

### Three Ollama Models
- `stratos-scorer-v2` (8.7GB Q8_0) — fine-tuned scoring model
- `qwen3:30b-a3b` (~18GB) — inference model for strat agent chat and market analysis
- `qwen3:14b` (~9GB) — wizard model for briefings, profile generation

Scorer (8.7GB) + wizard/14b (~9GB) can coexist in 24GB VRAM. The 30B cannot coexist with anything.

## CRITICAL DISCOVERY (from cross-Claude review)

**The V2 baseline numbers are NOT comparable.** The 0.393 MAE (contaminated eval) was measured via **direct PyTorch bf16 inference** during Phase 3 of training. The 1.533 MAE (holdout eval) was measured via **Ollama serving a Q8_0 GGUF with a broken Modelfile** (with `/nothink`, `num_predict=2048`). These used completely different pipelines.

The 1.533 holdout number conflates FOUR variables:
1. **Generalization gap** — unseen articles vs memorized articles
2. **Quantization overhead** — bf16 → Q8_0 GGUF
3. **Format drift** — `/nothink` causing 41% parse failures, prompt changes
4. **Ollama serving layer** — different inference engine than training eval

**We have never measured holdout performance with a correct pipeline.** The true V2 holdout MAE is unknown and could be anywhere from 0.7 to 1.5.

|----------|----------|--------|-----------------|
| Contaminated | Direct PyTorch bf16 | **DONE: 0.393 MAE** | Memorization ceiling |
| Contaminated | Ollama Q8_0 + correct Modelfile | **NOT DONE** | Quantization + serving cost |
| Holdout | Direct PyTorch bf16 | **NOT DONE** | True generalization |
| Holdout | Ollama Q8_0 + correct Modelfile | **NOT DONE** (1.533 was with broken format) | Real-world deployment |

## 4. V2 Problems Diagnosed Before V2.1 Training

### 4.1 — `/nothink` in Modelfile caused 41% parse failures
The V2 Modelfile included a `/nothink` directive to suppress Qwen3 think blocks. This caused the model to enter degenerate token loops, outputting slash-commands (`/soft`, `/nostart`, `/assistant`, `/nostart`...) instead of scores. **41% of outputs were unparseable.** Removing `/nothink` and letting Qwen3 produce natural empty think blocks `<think>\n</think>` dropped parse failures to ~5%. The think blocks are ~10 tokens of overhead and get stripped by the parser.

### 4.2 — Chinese output from Qwen3 (~7% parse failures)
The Qwen3 base model sometimes outputs Chinese score format: `力评分：3.5 | 原因：Article about...` instead of `SCORE: 3.5 | REASON: ...`. The V2 parser only matched the English format, causing ~7% additional parse failures. Combined with /nothink, total V2 parse failures reached 7.2% on holdout eval (108/1500 samples lost).

### 4.3 — Mid-range scoring collapse (moderate MAE 2.77, high MAE 2.56)
V2's worst performance was on scores 4.5-8.5 (moderate and high bands). The model had a strong tendency to **default to 8.5** for any article it found somewhat relevant, instead of discriminating between 5.0, 6.5, 7.0, 8.0, etc. Root cause: training data only contained short 1-sentence reasons (~154 chars) that didn't teach the model *why* articles fall in specific score ranges. The model learned "relevant = high score" without nuance.

### 4.4 — Training used only short `reason` field, discarding `think_text`
Opus scoring generates two fields:
- `reason`: Short 1-sentence summary (~154 chars avg)
- `think_text`: Full multi-paragraph reasoning (~1,133 chars avg)

V2 training only used the short `reason`. The 7.3x longer `think_text` was generated, paid for, and discarded.

### 4.5 — Eval contamination (same articles in train and eval)
The original `eval_v2.jsonl` used the same 813 articles as training (just different profiles). A proper holdout set with 50 completely unseen articles was created (`eval_holdout_v2.jsonl`), revealing the true performance gap.

### 4.6 — Only 30 profiles (limited diversity)
Missing common professions like teachers, truckers, dentists, pilots, electricians. This limited generalization.

### 4.7 — Post-training format drift
After V2 training, code changes to `scorer_adaptive.py` prompt format caused training/inference misalignment. MAE degraded from 0.393 to 0.761 on contaminated eval.

### 4.8 — No keyword pre-filter (100% LLM scoring)
Every article went through the LLM scorer, including obvious noise. Wasted ~50-60% of LLM inference time.

## 8. Key V2.1 Change: Think_text in Training

### What changed
- **V2 (before):** Assistant message used short `reason` field (~172 chars avg, max 313)
  - Example: `SCORE: 1.0 | REASON: Article about pediatric cancer is unrelated to petroleum engineering interests.`
- **V2.1 (now):** Assistant message uses full `think_text` field (~843 chars avg) when available
  - Example: `SCORE: 1.0 | REASON: Looking at this user's profile, they are a Petroleum Engineering student at Kuwait University with interests in reservoir simulation, drilling tech, and oil/gas careers. The article discusses pediatric oncology treatment advances at King Faisal Hospital in Riyadh. While both are in technical/scientific domains, there is zero overlap between oncology research and petroleum engineering. The user tracks companies like KOC, SLB, and Halliburton — none of which are mentioned. The geographic relevance to the Gulf region is the only tangential connection, but that alone does not make an oncology article useful to a petroleum engineering student.`

### Statistics
- 92.2% of training examples (17,511) use think_text (>200 chars)
- 7.8% of training examples (1,489) use short reason (<200 chars)
- Average assistant message: 843 chars (was 172 in V2) — **4.9x increase**
- Max: 1,798 chars, estimated ~450 tokens max (fits in 1024 seq length)

### Rationale
V2's worst performance was on mid-range scores (moderate band MAE 2.77, high band MAE 2.56). The model would default to 8.5 when uncertain. By training on Opus's full reasoning, the model should internalize *why* articles score in specific ranges, not just memorize score patterns.

### Quality Concern: Agent vs Opus Think_text
| Source | Avg think_text length | Quality |
|--------|----------------------|---------|
| Opus (Batch API) | 1,133 chars | Genuine nuanced reasoning, considers profile deeply |
| Claude Code agents | 269 chars | Formulaic, template-based, keyword-matching style |

**Opus example:** *"Looking at this user's profile, they are a Computer Engineering student in Kuwait with strong interests in embedded systems and seeking their first engineering position. The article discusses Innovobot's acquisition of Voler Systems, a California-based engineering firm specializing in embedded systems, sensor integration, and IoT solutions..."*

**Agent example:** *"Article 'Innovobot Acquires Voler Systems' from The Indianapolis Star. Profile is a Marketing director at a consumer goods company in London, UK. The article's subject matter falls outside this profile's professional scope and personal interests."*

**Risk:** 14,684 agent-sourced examples (37.6% of total scores) may teach the model formulaic reasoning patterns rather than genuine analysis.

## 9. Other V2.1 Changes

### Chinese Parse Fix
Qwen3 sometimes outputs Chinese instead of English: `力评分：3.5 | 原因：...` instead of `SCORE: 3.5 | REASON: ...`. Added regex fallback `SCORE_RE_ZH = re.compile(r"(?:力评分|评分)[：:]\s*(\d+\.?\d*)")` in both `scorer_adaptive.py` and `evaluate_scorer.py`.

### Keyword Pre-filter (Phase 1.5)
Before sending articles to LLM, checks keyword overlap between article text and profile interests. Zero keyword matches + low rule score → score as noise (skip LLM). Saves ~50-60% of LLM calls for obvious noise articles.

### Modelfile Fix
Removed `/nothink` directive that was causing 41% parse failures. Now Qwen3 produces natural empty think blocks `<think>\n</think>` which get stripped by the parser.

### Inference Speed Fix
Reduced `num_predict` from 2048 → 256 in `scorer_base.py`. The SCORE token comes first in model output, so we only need ~50-100 tokens. Prevents the model from generating verbose responses. Modelfile also needs updating from 2048 → 256 before deployment.

## 11. Known Pitfalls & Gotchas

### PEFT Meta Device Fix
After `get_peft_model()`, you MUST delete `hf_device_map` and force `.to("cuda:0")`. Otherwise gradients crash during training on ROCm.

### Adapter Inference is 30x Slower
Running inference through DoRA adapters on ROCm is ~30x slower than through a merged model. **Always merge adapters before bulk eval or deployment.** Phase 3 of `train_v2.py` handles this automatically.

### Log Buffering
Phase 3 output may not appear in log file immediately due to stdout buffering. Use `tee` in the training command.

### Qwen3 Think Blocks
- Do NOT set `"think": False` in Ollama API calls — Qwen3 ignores it and leaks reasoning as plain text into `content`
- Do NOT use `/nothink` in Modelfile — causes 41% parse failures from slash-command loops
- Let Qwen3 produce natural empty think blocks — they get stripped by `_call_ollama()` in scorer_base.py
- Code handles: closed think blocks (regex strip), unclosed/truncated think blocks (strip + warn), think-only responses (fallback score extraction)
- **Risk with V2.1:** If model outputs `<think>[reasoning]</think>` BEFORE `SCORE:` and total exceeds `num_predict`, the score gets truncated. Smoke test after GGUF export will confirm.

### Ollama API Endpoint Inconsistency (verified March 6, 2026)
The production scorer uses a DIFFERENT Ollama endpoint than everything else:

| Component | Endpoint | Stream | Temp | num_predict |
|-----------|----------|--------|------|-------------|
| **Scorer (production)** `scorer_base.py:828` | `/api/generate` | Yes | 0.6 | 256 |
| **Eval script** `evaluate_scorer.py:64` | `/api/chat` | No | 0.1 (flag: --temperature) | 512 |
| **Briefing** `briefing.py:166` | `/api/chat` | No | 0.7 | varies |
| **Agent chat** `agent.py:491` | `/api/chat` | Yes | varies | varies |
| **Wizard/profile gen** | `/api/chat` | No | varies | varies |
| **Server warmup** `server.py:1143` | `/api/generate` | No | N/A | 1 |

**The scorer is the ONLY production component using `/api/generate`.** Everything else uses `/api/chat`.

**Token sequences are identical** — the Modelfile TEMPLATE produces the same ChatML tokens (`<|im_start|>system...`) whether called via `/api/generate` with `prompt`+`system` or via `/api/chat` with `messages`. So the model sees the same input either way.

**The real differences:**
1. **Think block handling** — `/api/generate` returns think blocks inline in the `response` field (regex-stripped by `_call_ollama`). `/api/chat` may separate them into a `thinking` field, but `evaluate_scorer.py` still regex-strips from `message.content` as a safety net.
2. **Eval uses different settings than production** — temperature 0.1 vs 0.6 (configurable via `--temperature` flag), num_predict 512 vs 256 (eval higher for think block headroom), no streaming. Run eval at both temps for complete picture.

**Not urgent for V2.1 but worth fixing for V3:** Switch `_call_ollama` from `/api/generate` to `/api/chat` for consistency with the rest of the codebase and cleaner think block separation.

### num_predict Values
- **Production scorer_base.py:** 256 — SCORE comes first, 256 tokens is plenty for V2
- **Eval evaluate_scorer.py:** 512 — higher to give think block headroom (Ollama counts think tokens against num_predict)
- **Modelfile:** 256 — matches production. Template also fixed: `{{- end }}` on its own line with newline before `<|im_start|>user`
- **If V2.1 smoke test shows think blocks consuming >200 tokens:** bump production to 512 too

### Format Drift Detection
`stage4_prepare.py` must produce system/user/assistant messages matching `scorer_adaptive.py` inference format. Any code change to either file risks misalignment. Version-pinning prompt templates is a future improvement.

## 14. Findings from Cross-Claude Review

A separate Claude instance (which built the original V2) reviewed this document and provided analysis.

### Confirmed Facts
1. **V2's 0.393 MAE was direct PyTorch bf16 inference** — never went through Ollama/GGUF. The holdout 1.533 went through Ollama with broken Modelfile. These are incomparable measurements.
2. **V2 training data used short reasons** — verified: 172 chars avg, max 313 chars. V2.1's 843 chars avg is a 4.9x increase, not incremental.
3. **eval_v2.jsonl was split from same stage4 run** — identical format to training data. 0.393 tests memorization + format match, not generalization.
4. **V2 DID use dual weighting** — both WeightedRandomSampler AND per-sample loss weights. V2.1 preserves this correctly.
5. **Holdout is the metric that matters for shipping.** Contaminated metrics measure memorization, not production performance.
6. **V1 had CoT reasoning and got 1.553 MAE** — V2 improved via data scale (5,679→18,502 examples) and profile diversity (10→30), NOT via reasoning format. Think_text may not be the right lever for generalization.

### Open Risks
1. **Think_text might increase memorization** — longer reasoning repeats article-specific details in the training target, potentially teaching the model to pattern-match specific content rather than generalizable methods.
2. **Agent data (37.6% of scores) has formulaic reasoning** — 269 chars avg vs Opus 1,133 chars. May dilute training signal.
3. **813 articles may be the real bottleneck** — more diverse articles could matter more than richer reasoning on the same articles. DB has 16,477 untapped articles.
4. **The contaminated-holdout gap ratio matters** — V2: 0.393/1.533 = 3.9x reliance on memorization. If V2.1 has a smaller ratio (e.g. 1.0/1.2 = 1.2x), that indicates better generalization even if absolute contaminated MAE is worse.

### Consensus Recommendation
Let V2.1 finish. Evaluate on holdout. Also re-eval V2's original GGUF with corrected Modelfile on holdout to get a fair baseline. If V2.1 beats corrected-V2 on holdout, ship it. If not, prioritize expanding article diversity (813→2000+) over further format changes.

## 15. Open Questions

1. **Is training on think_text the right approach?** We're teaching the model to output verbose reasoning in hopes it internalizes better scoring logic. But at inference we cap output at 256 tokens and only parse the SCORE. Is the model learning useful representations, or just learning to be verbose?

2. **Agent data quality contamination:** 37.6% of total scores come from agents with formulaic 269-char think_text (vs Opus's 1,133-char genuine reasoning). Are these helping or hurting? Should we filter them out or weight them lower?

3. **90.8% noise in training data** — Even with weighted sampling (noise at 0.5x, critical at 3.0x), the model sees overwhelming majority 0-2 scores. Only 101 critical examples (0.5%). Is this enough signal for high-relevance scoring?

4. **Category always "general" in training** — At inference, the model sees actual categories like "AI & Technology" but was trained with Category: general for every example. Does this create a distribution shift?

5. **1 epoch, 1e-5 LR, cosine decay** — Loss plateaued around step 250 at ~5.0. Would a second epoch help? Higher LR? Different schedule?

6. **Loss values seem high** — Starting at 16.0, currently at 4.7. V2 started at ~2.2. Is this purely the longer assistant messages, or a sign of something wrong?

7. **Training on V1-merged base** — We're fine-tuning a model that already has V1 DoRA adapters merged. Is stacking fine-tunes like this risky for forgetting or instability?

8. **num_predict 256 at inference** — Will the model's learned verbosity cause it to "waste" tokens on preamble before SCORE, potentially getting truncated before outputting the score? (Smoke test planned.)

9. **Format misalignment** — Training uses `Category: general` and `Keywords: {profile interests}`, but inference uses actual category labels and category-specific keywords. Is this significant?

10. **Score distribution skew** — Average training score is 1.26. The model might develop a strong prior toward low scores. Is the dual weighting system sufficient to counteract this?
