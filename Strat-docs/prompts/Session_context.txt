‚ĚĮ What makes each step take so long? What are we exactly doing in each step?                                                                                                                                                                                  
                                                                                                                
‚óŹ Each step processes 16 examples (batch_size=2 √ó grad_accum=8) through this sequence:                                                                                                                                                                        
                                                                                             
  1. Forward pass ‚ÄĒ The full 8B parameter Qwen3 model processes each example's tokenized prompt+completion (up to 1024 tokens). That's matrix multiplications through 36 transformer layers, each with attention heads, FFN blocks ‚ÄĒ all in bf16 on the 7900  
  XTX                                                                                                                                                                                                                                                         
  2. Loss computation ‚ÄĒ Cross-entropy loss on the completion tokens only (the SCORE: X.X | REASON: ... part), weighted by the sample's band weight (noise=0.5x, critical=3.0x)                                                                                
  3. Backward pass ‚ÄĒ Backpropagation through the entire model to compute gradients. Even though only DoRA's 45M parameters (0.5% of 8B) are trainable, gradients still flow through the frozen layers to reach the DoRA adapters in q/k/v/o/gate/up/down_proj
  4. Gradient accumulation ‚ÄĒ Steps 2-3 repeat 8 times (grad_accum=8) before updating weights. Each micro-batch is 2 examples, so we do 8 forward+backward passes per optimizer step
  5. Optimizer step ‚ÄĒ AdamW updates the 45M DoRA parameters using the accumulated gradients, plus cosine LR scheduling
  6. Gradient checkpointing ‚ÄĒ To fit in 24GB VRAM, activations are recomputed during backward pass instead of stored. Saves ~40% memory but adds ~30% compute time

  So each "step" is really 8 forward passes + 8 backward passes through an 8 billion parameter model, with gradient checkpointing adding recomputation overhead. That's why it's ~32 seconds ‚ÄĒ it's doing a lot of math on a lot of parameters, trading speed
  for memory to fit on a single 24GB GPU.

Let me go through each one honestly:                                                                                                                                                                                                         
                                                                                                                                                                                                                                                              
  Whole vs Diff format                                                                                                                                                                                                                                        
  - "Whole" = training on complete files/outputs. "Diff" = training on just the changes (like git diffs)
  - Not useful for us. Our training examples are already minimal ‚ÄĒ SCORE: X.X | REASON: brief explanation. There's nothing to diff against. This matters more for code generation models that edit existing files.                                            
                                                                                                                
  Public Datasets (The Stack V2, Git datasets)
  - Large-scale code/text datasets used for pretraining or general fine-tuning
  - Not useful. Our task is hyper-specific ‚ÄĒ scoring article relevance for individual user profiles. No public dataset exists for this. Our value comes from Opus-generated ground truth tailored to our 30 profiles.

  Synthesizing Data / Synthetic Generation
  - Using a stronger model to generate training data for a weaker model
  - This is literally what we're already doing. Opus scores articles ‚Üí becomes training data for Qwen3-8B. Our entire pipeline is synthetic data generation. The distillation loop (Opus re-scores ‚Üí disagreements ‚Üí retrain) is textbook synthetic data.

  Data Augmentation
  - Creating variations of existing examples to increase diversity
  - Potentially useful. We could:
    - Paraphrase article titles/summaries and re-score them
    - Swap profiles for the same article (we already do this ‚ÄĒ 30 profiles √ó each article)
    - Perturb scores slightly to teach the model about score boundaries
  - But honestly, expanding with real new articles (what we just did) is better than artificial augmentation for our case. Augmented data risks teaching the model shortcuts.

  Magicoder / OSS-Instruct style generation
  - Using real-world artifacts (open source code) as seeds to generate training instructions, rather than purely synthetic prompts
  - Partially what we do. We use real articles from the live DB as seeds, then Opus generates the scoring. The key insight ‚ÄĒ grounding synthetic data in real artifacts ‚ÄĒ is already baked into our pipeline.

  Adding Reasoning to Data
  - Including chain-of-thought or explanations in training examples
  - Interesting possibility. Right now our training target is just SCORE: X.X | REASON: one sentence. We could have Opus generate detailed reasoning (why this score for this profile), then train the model on that richer output. This could improve score
  accuracy on ambiguous cases (the 3-7 range where we're weakest). The tradeoff: longer completions = slower inference = slower scanning.

  collaborative filtering at the individual level ‚ÄĒ the model provides the base score, user behavior provides the correction.

  4. What You're Actually Building

  Here's the thing I keep coming back to: the scorer is the core IP, but the product is the briefing, not the feed.

  A ranked list of articles is what every news app does. What makes StratOS "strategic intelligence" is the ability to say: "Three things happened this week that together mean X for your career/industry." That's the briefing system, and right now it's a
  post-processing afterthought.

  If I were building this as a product, the hierarchy would be:
  1. Daily briefing ‚ÄĒ the main thing users open the app for. 3-5 key insights synthesized across all articles
  2. Alert system ‚ÄĒ critical-scored articles push-notified immediately (not waiting for a scan cycle)
  3. Full feed ‚ÄĒ the ranked article list, for users who want to browse

  The scorer still powers all of this, but the value delivery changes.

  5. Profile Evolution

  Profiles are static YAML files. But real users' interests shift. Someone tracking "AI startups" six months ago now cares about "AI regulation."

  The data to detect this exists ‚ÄĒ user feedback patterns, click-through rates, which articles get retained. You could have the system suggest profile updates: "You've been engaging more with policy/regulation articles. Add 'AI governance' to your
  interests?"

  6. The Contamination Fix ‚ÄĒ Concretely

  This is the most immediate technical win. Here's what I'd do:

  - Reserve 50 articles that exist in NO training data
  - Score them with all 30 profiles via Opus ‚Üí 1,500 held-out examples
  - This becomes your permanent eval set, never touched by training
  - Cost: ~$1.50 via Batch API
  - Now you know your REAL accuracy, and every future training run can be compared fairly

  7. Robustness / Edge Cases

  Things that could break the scorer in production:
  - Clickbait titles ‚ÄĒ "You Won't Believe What Happened to Tesla" scores high on keywords but is garbage
  - Paywalled articles ‚ÄĒ summary is just the first paragraph, not enough to score accurately
  - Non-English content ‚ÄĒ you have language filtering but it's pattern-based
  - Duplicate stories ‚ÄĒ same event covered by 15 sources, all scored individually. User sees 15 versions of the same story ranked differently
